# Codex Plan Review

Claude Code の plan モードで生成された実装計画を、Codex MCP サーバー経由でレビューするスキル。コードベースも参照し、計画の妥当性を検証する。指摘の評価・修正・再レビューのループを自動で行う。

## フロー

### ステップ 1: plan ファイルの特定

- ユーザーが特定ファイルを指定していればそれを使用する
- 会話内で直前に作成・参照された plan ファイル名があればそれを優先する
- 上記いずれでも特定できなければ `ls -t ~/.claude/plans/*.md` で候補を取得する
  - 候補が1件 → そのファイルを使用
  - 候補が複数 → ファイル名を一覧表示し、ユーザーに選択を求める

### ステップ 2: PBI（要件）の取得

以下の優先順位で PBI を取得する:

1. 会話内で参照された要件ファイルやプロンプト（ユーザーが plan 作成時に指定した PBI）があればそれを使用する
2. 上記がなければ、Read ツールで plan ファイルを読み込み、以下のパターンで PBI への参照を探す:
   - ヘッダー部の `**Spec:**` や `**PBI:**` に続く `@docs/...` や `docs/...` 形式のパス参照
   - 見出しに requirements / spec / PBI / 要件 / 仕様 を含むセクション内の `docs/` 配下の Markdown ファイルパス
   - 抽出対象は `docs/` 配下の Markdown ファイル（`.md`）のみ。`src/` 等のコードファイルは無視する
   - 該当するファイルがあれば Read ツールで読み込み、PBI として使用する
3. いずれでも取得できなければ PBI なしで続行する（技術レビューのみ）

### ステップ 3: Codex MCP でレビュー実行

`mcp__codex__codex` ツールを呼び出してレビューを実行する。plan と PBI はファイルパスで指定し、Codex 自身に読み込ませる。Codex にはコードベースへのアクセス権があるため、plan 内で言及されているファイルやモジュールも実際に確認してレビューさせる。

**PBI ありの場合:**

````
mcp__codex__codex({
  prompt: "以下のファイルを読み込み、実装計画を要件定義およびコードベースと照らし合わせてレビューしてください。\n\n- 実装計画: <plan ファイルの絶対パス>\n- 要件定義: <PBI ファイルの絶対パス>\n\n計画内で言及されているファイルやモジュールも実際に読み込んでください。\n\n以下の実装計画を、要件定義と照らし合わせて厳密にレビューしてください。\n目的は、実装前に計画の品質を上げ、手戻り・抜け漏れ・設計事故を減らすことです。\n\nレビュー観点:\n1. 要件・完了条件を満たした計画になっているか\n2. 実装順序と依存関係に無理がないか\n3. 技術的な問題点、抜け漏れ、曖昧さがないか\n4. 既存機能への回帰リスクが考慮されているか\n5. テスト計画が十分か\n6. データ移行、ロールバック、運用監視の考慮が必要なのに漏れていないか\n7. セキュリティ、性能、権限、障害時対応の観点で不足がないか\n8. スコープが過大ではないか。より安全な分割や段階導入の余地がないか\n\nレビュー方針:\n- 抽象論ではなく、計画のどの部分をどう直すべきか具体的に指摘してください\n- 指摘は「実装時に事故や手戻りにつながるもの」を優先してください\n- 軽微な好みよりも、正しさ・安全性・保守性・実行可能性を優先してください\n- 要件に書かれていない機能追加は原則として要求せず、必要なら「要件確認事項」として分けてください\n\n回答は必ず以下の JSON 形式のみで返してください（それ以外のテキストは含めないでください）:\n{\n  \"overview\": {\n    \"verdict\": \"LGTM\" または \"NEEDS_FIX\" または \"BLOCKED\",\n    \"summary\": \"2〜4文の要約\"\n  },\n  \"findings\": [\n    {\n      \"id\": \"文字列\",\n      \"title\": \"短い見出し\",\n      \"severity\": \"critical/high/medium/low\",\n      \"category\": \"requirements/sequencing/technical/regression/testing/migration/security/performance/operations/scope\",\n      \"problem\": \"何が問題か\",\n      \"impact\": \"そのまま進めると何が起こるか\",\n      \"suggested_change\": \"計画にどう追記・修正すべきか\",\n      \"affected_step\": \"該当する計画のステップ名や箇所\"\n    }\n  ],\n  \"requirement_questions\": [\"要件や前提が曖昧で実装前に確認すべき事項\"],\n  \"suggested_diff\": \"元の計画をどう直すべきか、追記案の形で簡潔に\"\n}\n\nfindings が無い場合は空配列、requirement_questions が無い場合も空配列にしてください。"
})
````

**PBI なしの場合:**

````
mcp__codex__codex({
  prompt: "以下のファイルを読み込み、実装計画をコードベースと照らし合わせてレビューしてください。\n\n- 実装計画: <plan ファイルの絶対パス>\n\n計画内で言及されているファイルやモジュールも実際に読み込んでください。\n\n以下の実装計画を厳密にレビューしてください。\n目的は、実装前に計画の品質を上げ、手戻り・抜け漏れ・設計事故を減らすことです。\n\nレビュー観点:\n1. 実装順序と依存関係に無理がないか\n2. 技術的な問題点、抜け漏れ、曖昧さがないか\n3. 既存機能への回帰リスクが考慮されているか\n4. テスト計画が十分か\n5. データ移行、ロールバック、運用監視の考慮が必要なのに漏れていないか\n6. セキュリティ、性能、権限、障害時対応の観点で不足がないか\n7. スコープが過大ではないか。より安全な分割や段階導入の余地がないか\n\nレビュー方針:\n- 抽象論ではなく、計画のどの部分をどう直すべきか具体的に指摘してください\n- 指摘は「実装時に事故や手戻りにつながるもの」を優先してください\n- 軽微な好みよりも、正しさ・安全性・保守性・実行可能性を優先してください\n\n回答は必ず以下の JSON 形式のみで返してください（それ以外のテキストは含めないでください）:\n{\n  \"overview\": {\n    \"verdict\": \"LGTM\" または \"NEEDS_FIX\" または \"BLOCKED\",\n    \"summary\": \"2〜4文の要約\"\n  },\n  \"findings\": [\n    {\n      \"id\": \"文字列\",\n      \"title\": \"短い見出し\",\n      \"severity\": \"critical/high/medium/low\",\n      \"category\": \"requirements/sequencing/technical/regression/testing/migration/security/performance/operations/scope\",\n      \"problem\": \"何が問題か\",\n      \"impact\": \"そのまま進めると何が起こるか\",\n      \"suggested_change\": \"計画にどう追記・修正すべきか\",\n      \"affected_step\": \"該当する計画のステップ名や箇所\"\n    }\n  ],\n  \"requirement_questions\": [\"要件や前提が曖昧で実装前に確認すべき事項\"],\n  \"suggested_diff\": \"元の計画をどう直すべきか、追記案の形で簡潔に\"\n}\n\nfindings が無い場合は空配列、requirement_questions が無い場合も空配列にしてください。"
})
````

**注意事項:**
- plan と PBI はファイルパス（絶対パス）で指定する。内容のインライン展開は行わない
- `approval-policy` は指定しない（Codex がファイルを読み取れるようにするため）

**パースと threadId の記録:**
1. `mcp__codex__codex` の応答から JSON 部分を抽出してパースする
   - 応答全体が JSON でない場合は、`{` で始まる行から `}` で終わる行までを抽出する
   - パース失敗の場合: 1回だけ再試行する（同じプロンプトで MCP ツールを再呼び出し）
   - 再試行でもパース失敗の場合: Codex の応答をそのままユーザーに表示してフローを中断する
2. Codex がエラーを返した場合は、エラー内容をユーザーに報告してフローを中断する
3. **応答に含まれる `threadId` を記録しておく。** ステップ 5 の再レビューで同一セッションを継続するために必要

### ステップ 4: 指摘の評価と修正

1. JSON の `overview.verdict` が `"LGTM"` であれば → ステップ 6 へ
2. `overview.verdict` が `"BLOCKED"` の場合 → ブロッカーをユーザーに報告し、ユーザーの判断を仰ぐ
3. `requirement_questions` がある場合 → ユーザーに要件確認事項として提示する
4. `findings` 配列の各指摘を評価する:
   - 技術的に妥当か
   - plan のコンテキストに合っているか
   - PBI の要件に照らして正当か（PBI がある場合）
   - severity と impact の妥当性
5. 妥当と判断した指摘のみ、Edit ツールで plan ファイルを修正する。`suggested_diff` も参考にする
6. 却下した指摘は `id` と却下理由を記録する

### ステップ 5: 再レビューループ

- ループ回数をカウントする（最大5回）
- ステップ 3 で記録した `threadId` を使い、`mcp__codex__codex-reply` ツールで同一セッションを継続する。プロンプトには以下を含める:
  1. 修正内容の説明（どの指摘を採用し、どう修正したか）
  2. 却下した指摘の除外指示（却下した指摘がない場合は、このセクション自体を省略する）。形式:
  ```
  以下の指摘は検討済みです。これらについては再度指摘しないでください：
  - F1: <却下理由>
  - F3: <却下理由>
  ```
  3. 修正済みの plan ファイルを再度読み込んでレビューする指示

````
mcp__codex__codex-reply({
  threadId: "<ステップ 3 で記録した threadId>",
  prompt: "plan ファイルを修正しました。\n\n<修正内容の説明>\n\n<却下した指摘の除外指示（該当がある場合のみ）>\n\n修正済みの plan ファイルを再度読み込み、同じ観点・同じ出力形式でレビューしてください。\n\n- 実装計画: <plan ファイルの絶対パス>"
})
````

- ステップ 3 のパース手順と同様に JSON を抽出してパースする
- JSON の `overview.verdict` が `"LGTM"` であれば → ステップ 6 へ
- `"NEEDS_FIX"` であれば → ステップ 4 に戻る
- `"BLOCKED"` であれば → ステップ 4 に戻る（ブロッカー対応）
- 5回到達で `"LGTM"` にならなければ → ステップ 6 へ
- **重要:** 全指摘を採用した場合でも、`overview.verdict` が `"LGTM"` になるまでループを継続すること。独自判断で打ち切らないこと

### ステップ 6: 完了報告

最終結果をユーザーに報告する:
- 総評（verdict と summary）
- 採用した指摘と修正内容
- 却下した指摘と理由
- 要件確認事項（あれば）
- ループ回数
- 最終ステータス（LGTM / 5回到達で終了）

## 注意事項

- Codex にはコードベースへのアクセス権がある。plan 内で言及されているファイルやモジュールも実際に確認してレビューさせる
